# Blindspot Audit Workflow

Updated: 2026-05-08

Purpose:
- audit empirical output before interpretation hardens into prose
- find visible anomalies, missing checks, overlooked mechanisms, and undersold strengths
- prevent tables and figures from becoming `FIX` only because the main finding looks useful

Source:
- Adapted from Scott Cunningham's `MixtapeTools` Blindspot protocol: https://github.com/scunning1975/MixtapeTools/tree/main/skills/blindspot
- Operational upstream skill: https://github.com/scunning1975/MixtapeTools/blob/main/.claude/skills/blindspot/SKILL.md
- Source evidence grade: `B` for workflow use. It is a concrete public research-output audit protocol, but it is not manuscript literature evidence.
- Local deployment status: accepted by human instruction on 2026-04-30.

## ARIS Bridge

Read `01_policies/aris_bridge.md` before invoking ARIS skills inside this workflow.

Engine skills (run alongside this audit, never in place of it):
- `/experiment-audit [results-dir]` — cross-model integrity check that reads source code and result files directly. Catches fake ground truth, score normalization, phantom results, insufficient scope. Run before this blindspot audit when the underlying analysis was generated by an LLM agent or any pipeline whose evaluation honesty has not been independently verified.
- `/result-to-claim "experiment description"` — Codex MCP judges whether numerical results support intended claims. Useful before our Vice 2 (Convenient Absence) check, because the verdict surfaces missing robustness, placebo, and subgroup checks an external reviewer would expect.

Binding rules:
- Our four-quadrant audit (Vice 1 unexplained feature, Vice 2 convenient absence, Virtue 1 unasked question, Virtue 2 unexploited strength) is authoritative for econ/finance exhibits.
- ARIS skills cover engineering integrity (fraud, phantom results, scope). Our audit covers economic interpretation, mechanism, identification, and presentation.
- A `CLEAR` ARIS audit does not produce a `CLEAR` blindspot ruling on its own.
- A `HOLD` ARIS audit blocks `FIX` regardless of our blindspot ruling.
- Reviewer-independence rule applies: the model that ran or wrote the analysis must not also rule on integrity. Our author and the ARIS executor must be different model families from the ARIS reviewer.
- Cross-reference outputs: when both audits run, link the ARIS `EXPERIMENT_AUDIT.md` and `RESULT_TO_CLAIM.md` paths in our blindspot memo's "Output audited" section.

## Trigger

Use this workflow when:
- a table, figure, or result set first looks interpretation-ready
- writing kickoff needs to decide whether an exhibit can be `FIX`
- a surprising spike, sign flip, sample-size change, or null result appears
- the user asks what the output might be hiding
- a result seems too clean, too convenient, or too narrowly interpreted

Do not use this workflow to check implementation correctness. Use `02_workflows/referee_audit.md` for code, replication, output automation, directory, and econometrics audits.

## Required Inputs

Gather:
- output path
- source script and generation command
- table manifest, figure manifest, or mapped provenance
- current intended interpretation
- relevant sample, unit, period, model, controls, fixed effects, and standard errors
- prior exhibit memo if one exists

If an input is missing, record it as `[GAP: ...]`.

## Four-Quadrant Audit

### 1. Vice: Unexplained Feature

List all visible features before interpreting the main result:
- coefficients, signs, magnitudes, standard errors, and sample sizes
- spikes, dips, discontinuities, and outliers
- column-to-column changes
- panel-to-panel changes
- axis scale, binning, smoothing, and missing categories

For each feature, ask what could generate it:
- rounding or discretization
- sample restriction
- merge or missing-value rule
- measurement artifact
- small-sample noise
- timing mismatch
- substantive mechanism

Flag the hardest feature to explain under the preferred interpretation.

### 2. Vice: Convenient Absence

List what a skeptical reader would expect but cannot see:
- missing robustness checks
- missing placebo outcomes or periods
- missing subgroup or heterogeneity checks
- missing pre-trend, first-stage, balance, or diagnostic plot
- unexplained sample exclusion
- unexplained `N` changes across columns

Mark each absence as:
- `Not needed`
- `Needed before FIX`
- `Needed before submission`
- `Cannot assess`

### 3. Virtue: Unasked Question

Look for patterns that may deserve a better question:
- heterogeneity richer than the average effect
- mechanism evidence not yet used
- secondary outcomes that explain the main result
- descriptive facts that deserve a figure or table
- appendix results that belong in the main paper

Record whether each opportunity changes:
- research question
- exhibit plan
- mechanism section
- robustness section
- future-project parking lot

### 4. Virtue: Unexploited Strength

Look for design or evidence strength the current interpretation undersells:
- stronger source of variation than the prose claims
- falsification test already available in the data
- diagnostic that directly answers a likely objection
- cleaner visual display than the current table or figure
- broader literature positioning supported by the result

Do not inflate claims. Strengths become usable only when the source evidence and identification support them.

## Output

Default path:

```text
paper/exhibit_memos/blindspot_<exhibit_id>.md
```

For analysis batches before writing kickoff:

```text
output/diagnostics/blindspot/<YYYY-MM-DD>_<slug>.md
```

Required sections:
- Output audited
- Intended interpretation
- Vice 1: Unexplained feature
- Vice 2: Convenient absence
- Virtue 1: Unasked question
- Virtue 2: Unexploited strength
- Ruling
- Required changes before interpretation

Ruling labels:
- `CLEAR`: proceed to interpretation; record noted virtues
- `CONDITIONAL`: proceed only with explicit caveats or follow-up actions
- `HOLD`: do not use this output for final interpretation until the project resolves flagged vices

## Integration With Exhibit Gates

Before an exhibit can be labeled `FIX`, its exhibit memo must either:
- include a completed blindspot audit, or
- state why blindspot audit does not apply

A `HOLD` ruling blocks `FIX`.

A `CONDITIONAL` ruling can support `FIX` only when the memo states the caveat, exact claim boundary, and follow-up status.

Update `paper/writing_kickoff_decisions.md` with:
- blindspot report path
- ruling
- required interpretation boundary
- unresolved follow-up if any

## Guardrails

- Do not treat blindspot findings as proof of error without checking provenance.
- Do not add new claims only because they are interesting.
- Do not bury a vice in prose; resolve it, caveat it, or mark the exhibit `REVISE` or `GAP`.
- Do not use this workflow as a substitute for referee audit.
